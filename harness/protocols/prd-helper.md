# PRD Helper 프로토콜

`$prd-helper`는 PRD를 함께 작성하는 보조 단계다. PRD는 구현 지시서가 아니라
"왜 이 작업을 하는가, 무엇이 완료인가, 무엇은 하지 않는가"를 보존하는 제품
아티팩트다. 이 단계는 세 흐름을 엮어 PRD 품질을 끌어올린다.

- `$deep-interview`로 사용자 의견과 결정 경계를 수집한다.
- `researcher` 역할로 레퍼런스/선행 사례/참고 자료를 수집한다.
- `reviewer` 역할로 작성 중인 PRD를 지속 검토한다.

전제: `$kickoff`로 raw unit(`docs/raw/<type>/<slug>/`)이 이미 생성돼 있고, 그 안에
단계 체크포인트 원장 `state.md`가 있다. PRD는 `review` 상태로 작성하고, 사용자 명시
승인 전에는 `approved`로 바꾸지 않는다. `approved` 전환은 오직 `harness:approve`로만
한다(직접 frontmatter 편집 금지 — 런타임 훅과 `harness:check`가 막는다).

## 언어

PRD는 한국어로 작성한다. branch name, file path, command, TypeScript identifier처럼
기계가 읽거나 코드와 직접 연결되는 표현만 영어를 유지한다.

## 진행 루프

### Phase 1: 의견 수집

1. 목표, 성공 기준, 비목표, 결정 경계를 끌어낸다. `$deep-interview`가 설치돼
   있으면 그 스킬을 우선 사용하고, 질문 transport는 deep-interview 내부에서 현재
   surface에 맞게 선택한다. `$deep-interview`가 없을 때만 현재 런타임의 구조화
   질문 도구로 직접 질문하고, 구조화 질문 도구가 없을 때만 한 번에 하나씩 간결한
   명시 질문으로 fallback한다.
2. 사용자가 직접 판단해야 하는 항목은 숨은 가정으로 처리하지 않고 먼저
   `$deep-interview` 또는 현재 런타임의 구조화 질문 도구로 묻는다. 둘 다 없을
   때만 명시 질문한다.

### Phase 2: 레퍼런스 수집 (`researcher`)

1. `researcher` 역할로 유사 제품, 선행 사례, 표준, 제약을 조사한다.
2. 조사 결과는 출처와 함께 `notes.md`에 정리하고, PRD에는 결론만 반영한다.
3. 조사가 PRD의 어떤 요구/비목표/수용 기준을 뒷받침하는지 연결한다.

### Phase 3: 초안 작성

아래 필수 섹션으로 PRD를 작성한다.

```md
## 배경
## 목표
## 비목표
## 요구사항
### 기능 요구사항
### 비기능 요구사항
## 수용 기준
## 열린 질문
## ADR 필요 여부
## 관련 문서
```

작성 기준:

- 배경은 사용자 또는 프로젝트 관점의 긴장을 설명한다.
- 목표는 이 작업이 달성해야 하는 결과를 적는다.
- 비목표는 범위 팽창을 막기 위해 구체적으로 적는다.
- 기능 요구사항은 사용자가 경험하거나 개발자가 구현해야 하는 동작을 적는다.
- 비기능 요구사항은 성능, 접근성, 반응형, 저장소 구조, 검증 가능성 같은 품질 조건을 적는다.
- 수용 기준은 검증자가 통과/실패를 판단할 수 있어야 한다.
- 열린 질문은 숨은 가정으로 묻지 말고 명시한다.

### Phase 4: 지속 리뷰 (`reviewer`)

`reviewer` 역할로 초안을 반복 검토하고, 지적 사항을 반영해 다시 다듬는다.

- 수용 기준이 관찰 가능한가? (통과/실패를 판정할 수 있는가)
- 목표와 비목표가 서로 모순되지 않는가?
- 누락된 비기능 요구(접근성/반응형/저장소/검증)는 없는가?
- 구현 세부를 PRD에 과하게 박지 않았는가?
- 레퍼런스 결론이 요구사항과 실제로 연결되는가?

리뷰 → 수정 루프는 위 질문에 모두 답할 수 있을 때까지 반복한다.

### Phase 5: ADR 필요 여부 판단

`## ADR 필요 여부` 섹션에서 구조/데이터/엔진/의존성 같은 durable decision이
필요한지 판단한다.

- 필요하면 `$adr-helper`로 넘어가 ADR을 작성한다(ADR은 `proposed` 상태 초안).
- 필요 없으면 그 이유를 한 줄로 남긴다. ADR 작성은 선택이다.

ADR "작성"이 선택이라는 뜻이지 `adr.md` 파일은 선택이 아니다. feature 단위는 항상
`adr.md`를 유지한다(`$kickoff`가 생성하고 `harness:check`가 존재를 요구한다). ADR이
불필요하면 본문에 "불필요: <이유>" 한 줄만 남기고 status는 `proposed`로 둔다. 파일을
지우지 않는다.

## 정리와 승인

PRD 초안이 자리를 잡으면:

```sh
npm run harness:ingest -- docs/raw/<type>/<slug>
npm run harness:check
```

이 단계는 PRD를 `review` 상태로 마무리한다. `review`로 올릴 때 `state.md`의 `stage`를
`prd-review`로, `prd_status`를 `review`로 갱신하고 단계 로그에 한 줄 남긴다. 여기까지가
이 프로토콜의 끝이다. **여기서 절대로 approved로 넘어가지 않는다.**

### 승인은 오직 명시 의례로만

`approved` 전환은 사용자의 명시 승인이 있을 때만, 오직 `harness:approve`로만 한다.

1. 현재 런타임의 구조화 질문 도구(ClaudeCode는 `AskUserQuestion`)로, **대상 문서와 전환
   상태를 명시해** 승인을 요청한다. 예: "이 PRD를 approved로 전환할까요? (승인 / 아직)".
2. 사용자가 분명히 승인하면 **그 발화를 그대로 인용**해 실행한다.

```sh
npm run harness:approve -- --unit docs/raw/<type>/<slug> --quote "<사용자의 승인 발화 verbatim>" [--adr]
```

`harness:approve`가 status·`approval:` 근거·`state.md` 승인 이벤트를 원자적으로 함께
기록한다. 에이전트가 직접 frontmatter status를 `approved`로 고치지 않는다.

**승인으로 간주하지 않는 것(중요):** "이렇게 하려고 했어", "이거 작업해야 돼", "좋아
보인다", 목표·범위·아이디어 설명, 인터뷰 질문에 대한 답 — 이 중 어느 것도 승인이
아니다. 승인은 위 승인 요청에 대한 사용자의 분명한 긍정 응답만을 뜻한다. 조금이라도
모호하면 승인이 아니며, `review`로 둔 채 다시 구조화 질문으로 확인한다.

구현은 승인이 `state.md`에 기록된 이후 `feature-develop`에서 진행한다.

## 품질 게이트

PRD 초안은 아래 질문에 답할 수 있어야 한다.

- 왜 지금 이 작업인가?
- 이 작업이 끝났다고 어떻게 판단하는가?
- 어떤 일은 일부러 하지 않는가?
- 다음 작업 단위와 어떻게 연결되는가?
- 구현자가 설계 결정을 별도로 ADR에 남겨야 하는가?

## 실패 모드

- **나쁨:** "데이터를 만든다"처럼 선언만 있고 수용 기준이 없다.
- **좋음:** "필수 참조 데이터가 중복 없이 로드되고 누락 시 검증이 실패한다"처럼 관찰 가능한 기준을 둔다.

- **나쁨:** 레퍼런스 조사 없이 감으로 요구사항을 적는다.
- **좋음:** `researcher`로 선행 사례를 모으고 결론을 요구사항에 연결한다.

- **나쁨:** 초안을 한 번 쓰고 그대로 둔다.
- **좋음:** `reviewer` 리뷰 → 수정 루프로 수용 기준과 비목표를 다듬는다.

- **나쁨:** PRD 초안을 작성하자마자 `approved`로 둔다.
- **좋음:** `review`로 두고, 사용자 명시 승인 발화를 받아 `harness:approve`로 `approved`로 바꾼다.

- **나쁨:** "이렇게 하려고 했어" 같은 의도·아이디어 발화를 승인으로 오인해 바로 전환·구현한다.
- **좋음:** 그런 발화는 승인이 아니라고 보고, 구조화 질문으로 명시 승인을 다시 확인한다.
