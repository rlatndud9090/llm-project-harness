# ADR Helper 프로토콜

`$adr-helper`는 ADR을 함께 작성하는 보조 단계다. 구조는 `$prd-helper`와 거의
같다(의견 수집 → 레퍼런스 수집 → 초안 → 지속 리뷰). 다른 점은 다루는 대상이
"제품 요구"가 아니라 "구조 결정"이라는 것이다.

**ADR 작성은 선택이다.** 모든 작업에 ADR이 필요하지는 않다. ADR이 필요한지는
`$prd-helper`의 `## ADR 필요 여부` 단계에서 판단하고, 필요하다고 결론났을 때만
이 단계로 들어온다.

전제: feature raw unit에 `adr.md`가 있다(`$kickoff`가 생성). ADR은 `proposed`
상태로 작성하고, 사용자 명시 승인 전에는 `accepted`로 바꾸지 않는다.

## ADR이 필요한 경우

- 데이터 계약, 상태 모델, 저장소 스키마 같은 구조를 정한다.
- 엔진/모듈 경계, 의존성 도입처럼 되돌리기 어려운 선택을 한다.
- 대안이 여럿이고 기각 이유를 기록으로 남겨야 한다.

위에 해당하지 않으면 ADR을 만들지 않고 PRD에 그 이유를 한 줄 남긴다.

## 진행 루프

### Phase 1: 의견 수집 (`$deep-interview`)

결정해야 하는 선택지와 제약, 사용자가 직접 골라야 하는 경계를 끌어낸다.

### Phase 2: 레퍼런스 수집 (`researcher`)

`researcher` 역할로 대안별 선행 사례, 트레이드오프, 표준을 조사한다. 출처는
`notes.md`에 남기고 ADR에는 결론과 근거만 반영한다.

### Phase 3: 초안 작성

```md
## 배경
## 결정
## 대안
## 기각 이유
## 결과
## 검증 방법
## 관련 문서
```

- 결정은 채택한 구조 하나를 분명히 적는다.
- 대안은 최소 2개를 적고, 각각의 기각 이유를 남긴다.
- 결과는 이 결정이 만드는 제약과 후속 영향을 적는다.
- 검증 방법은 결정이 의도대로 동작하는지 확인할 방법을 적는다.

### Phase 4: 지속 리뷰 (`reviewer`)

`reviewer` 역할로 초안을 반복 검토한다.

- 대안이 형식적으로만 나열되지 않고 실제로 비교되었는가?
- 기각 이유가 납득 가능한가?
- 결정이 PRD의 요구/수용 기준과 충돌하지 않는가?
- placeholder 문장만 있고 실제 결정이 비어 있지 않은가?

## 승인과 불변성

ADR은 에이전트가 단독으로 `accepted` 처리하지 않는다. 사용자 명시 승인이 있을
때만 status를 전환하고 `approval: "user:YYYY-MM-DD:<근거>"`를 남긴다.

accepted ADR 본문은 과거 결정의 근거이므로 고쳐 쓰지 않는다. 결정이 바뀌면 새
ADR을 추가하고 옛 ADR을 `superseded`로 표시한다.

## 다음 단계

PRD/ADR이 모두 자리를 잡고 사용자 승인까지 끝나면 `feature-develop.md`로 넘어가
구현을 시작한다.

## 실패 모드

- **나쁨:** 모든 작업에 기계적으로 ADR을 만든다.
- **좋음:** durable decision이 있을 때만 ADR을 만들고, 아니면 PRD에 이유를 남긴다.

- **나쁨:** 대안 없이 결정 하나만 적는다.
- **좋음:** 대안 2개 이상과 기각 이유를 함께 남겨 결정 근거를 보존한다.

- **나쁨:** 에이전트가 ADR을 작성한 뒤 곧바로 `accepted`로 바꾼다.
- **좋음:** `proposed`로 두고, 사용자 승인 후 `approval:` 근거와 함께 `accepted`로 바꾼다.
