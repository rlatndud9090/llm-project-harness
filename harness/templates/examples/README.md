# PRD/ADR 고도 가이드 · 예시 문서

이 디렉터리는 소비 프로젝트가 **PRD/ADR을 올바른 고도(altitude)로 쓰도록** 돕는 교육용
참고 자료다. 빈 스켈레톤은 `../raw/`에 있고, 여기에는 **채워진 짝 예시**와 그 예시가 왜 이
고도인지 설명하는 **고도 루브릭**이 있다.

- [`notification-prd.md`](./notification-prd.md) — 채워진 PRD 예시(문제 공간, WHAT/WHY)
- [`notification-adr.md`](./notification-adr.md) — 그 PRD가 ADR 필요로 판정한 결정을 다루는
  짝 ADR 예시(해법 공간, 어떤 HOW를 왜 골랐나)

> 이 파일은 이 하네스의 **단일 출처(single source)** 다. `prd-helper`/`adr-helper` 프로토콜과
> `prd-writer`/`architect` 역할은 이 가이드를 **얇게 참조**만 하고 내용을 복제하지 않는다
> (복제는 drift를 낳는다). 고도 판정 규칙을 고칠 일이 있으면 여기만 고친다.

이 가이드는 특정 프레임워크·회사·제품에 매이지 않는 **보편 원칙**만 담는다. 도메인 특수성은
소비 프로젝트의 raw PRD/ADR 본문에 남긴다.

---

## 1. 무엇이 어디에 사는가 — 3층 고도

한 작업의 서술은 세 층으로 갈린다. **PRD는 문제 공간(왜/무엇)을, ADR은 해법 공간의 "어떤
방법을 왜 골랐나"를, 구현(코드·리뷰)은 상세 방법을 담는다.**

| 층 | 담는 질문 | 담기는 것 | 판별 규칙 |
|---|---|---|---|
| **PRD** (문제 공간, 왜/무엇) | 왜 하는가, 누구의 어떤 고통인가, 무엇이 완료인가 | 배경·문제, 측정 가능한 목표, 대상 사용자·유즈케이스, **구현 중립** 기능 요구(동작), 비기능 요구를 '무엇이 만족돼야 하는가'로, 명시적 비목표, **관찰 가능한** 수용 기준, 제약·가정, 열린 질문 | "관찰로 통과/실패를 판정할 수 있고, **특정 기술을 처방하지 않는가**" |
| **ADR** (해법 공간, 어떤 방법을 왜) | 여러 대안 중 무엇을 왜 골랐나 | 되돌리기 어려운 **결정 하나** + 컨텍스트(제약·힘)·선택지(기각안 포함)·선택 근거·결과(긍정·트레이드오프)·상태 | "**대안이 여럿이고, 되돌리기 비싸고**, 구조·데이터·경계·의존성에 영향을 주는가" |
| **구현** (코드·technical spec) | 내부적으로 어떻게 만드나 | 함수 시그니처, 변수·타입·필드 명명, 내부 알고리즘, 스키마 상세, 파라미터 값, UI 픽셀, 사소한 파일 배치 | "**바꿔도 사용자가 못 알아채고**, 대안 전환이 저비용인가" |

**코드레벨 디테일이 PRD에 새면 안 되는 이유:** (1) 엔지니어링의 해법 재량을 조기에 빼앗고
(premature commitment), (2) 구현이 바뀔 때마다 PRD가 함께 흔들려 유지보수가 무너지며,
(3) 코드를 읽어야만 판정 가능해져 수용 기준의 **관찰 가능성**이 깨진다.

## 2. 누수 판정 self-check (작성·리뷰 때 각 문장에 적용)

수용 기준·요구 문장마다 아래를 통과시킨다. 하나라도 걸리면 그 문장은 PRD 고도가 아니다.

1. **실제 요구 판정.** 이 문장은 "무엇이 만족돼야 하는가"인가, 아니면 "어떻게 만들지"(설계/
   구현 해법)인가? 후자면 PRD 고도가 아니다.
2. **구현 결합 판정.** 구현 방식이 바뀌면 이 문장도 바꿔야 하는가? 그렇다면 구현 세부가 샌
   것이니, 관찰 가능한 결과로 다시 쓰거나 ADR/구현으로 내린다.
3. **고유명사 소거 판정(LLM 특히 주의).** 이 문장에서 특정 언어·라이브러리·프레임워크·
   파일명·타입명·함수명·필드명·엔드포인트를 지우면 요구가 무너지는가? 무너지면 그 고유명사가
   구현 누수다. 지워도 남는 "무엇/왜"만 PRD에 남기고, 그 고유명사가 실제로 **다투는 결정**이면
   ADR로, 아니면 구현으로 내린다.

> **에이전트가 특히 빠지는 함정:** 코드베이스를 이미 읽은 상태에서 PRD를 쓰면, 방금 본
> 타입·파일·필드 이름을 그대로 요구에 옮겨 적기 쉽다. "구체적일수록 좋다"는 편향이 여기서
> 역효과를 낸다. 위 3번(고유명사 소거)이 이 누수를 겨냥한다.

**단, 구현을 빼려다 모호해지지 말 것.** 과소명세(vague)도 동등한 실패다. 목표는 "구현
중립적이되 관찰·통과/실패 판정 가능"이다. "빠르다/사용하기 좋다"가 아니라 "N개 항목이
중복 없이 로드되고, 누락 시 검증이 실패한다"처럼 쓴다.

## 3. 누수 부검 대조표 (나쁨 → 좋음)

실제로 자주 새는 문장 유형과, 같은 의도를 세 층으로 올바르게 나눈 재작성이다. 소재를 여러
도메인에서 뽑아 특정 도메인 편향을 피했다.

| 누수한 PRD 문장 (나쁨) | PRD 고도로 (좋음) | ADR로 내릴 것 | 구현으로 내릴 것 |
|---|---|---|---|
| "`NotificationEntity` 테이블에 `read_at: timestamp` 컬럼을 추가한다." | "사용자가 각 알림의 읽음/안읽음 상태를 확인할 수 있다." | (읽음 상태를 어디에 어떻게 저장할지 대안이 다투면) 저장 위치·모델 결정 | 컬럼명·타입 |
| "React Query로 15초마다 `/api/notifications`를 폴링한다." | "새 알림이 발생하면 사용자가 지연 없이 인지할 수 있다(허용 지연: 15초 이내)." | 전달 방식(폴링 vs 푸시)이 되돌리기 비싼 결정이면 ADR | 라이브러리·인터벌 값·엔드포인트 경로 |
| "검색 결과를 `Array.prototype.sort`로 관련도 내림차순 정렬한다." | "검색 결과가 사용자가 판단하기에 관련도 높은 순으로 상위에 온다." | 관련도 점수 산출 방식이 다투는 결정이면 ADR | 정렬 함수·구현 알고리즘 |
| "임포트 시 `papaparse`로 CSV를 파싱해 `ImportRow[]`로 매핑한다." | "사용자가 준비한 표 형식 파일을 올리면 각 행이 레코드로 반영된다." | 지원 포맷·중복 처리 정책이 결정거리면 ADR | 파서 라이브러리·타입 이름 |
| "빈 상태일 때 `#f5f5f5` 배경에 24px 아이콘을 중앙 정렬한다." | "데이터가 없을 때 사용자가 빈 상태임을 분명히 인지한다." | (빈 상태 화면 배치가 다투는 결정이면 디자인 결정 레인 ADR) | 색상 코드·픽셀·정렬 |

핵심: **왼쪽은 전부 "고유명사 소거 판정"에서 걸린다** — 타입명/라이브러리/컬럼명/색상코드를
지우면 요구가 무너지기 때문이다. 오른쪽은 지워도 "무엇/왜"가 온전히 남는다.

## 4. ADR 쪽 대칭 누수도 경계한다

누수는 PRD→구현 방향만 있는 게 아니다. **ADR에 코드 스니펫·상세 다이어그램·전체 설계문서를
욱여넣는 것**(과대 ADR)도 같은 병이다. ADR은 "이 결정을 왜 이렇게 했나"의 **근거 저널**이지
구현 매뉴얼이 아니다. 결정을 재현할 만큼의 컨텍스트·선택지·근거만 남기고, 상세 구현은 코드와
별도 설계문서에 둔다. 결정 하나당 ADR 하나를 지키고, accepted된 ADR 본문은 고쳐 쓰지 않는다
(결정이 바뀌면 새 ADR로 대체).

## 5. 더 읽을거리 (선택)

ADR 형식의 정평 난 공개 출처(도메인 중립, 안정 링크):

- Architecture Decision Records — https://adr.github.io/
- MADR (Markdown ADR) — https://adr.github.io/madr/
- Nygard 포맷 실물 ADR 예시(arc42) — https://docs.arc42.org/examples/decision-use-adrs/

PRD/요구 고도의 근거가 되는 보편 원칙(요구≠설계, 기능 명세 vs 기술 명세)은 요구공학·제품
문서화 문헌에 널리 확립돼 있다. 이 가이드의 판정 규칙은 그 보편 원칙을 하네스에 맞게 압축한
것이다.
