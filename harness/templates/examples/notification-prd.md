# PRD: 사용자 알림 목록

## 배경

사용자는 자신과 관련된 사건(누군가 내 항목에 반응함, 나에게 무언가 배정됨 등)을 제때
알지 못해 놓친다. 지금은 관련 화면을 사용자가 직접 다시 열어봐야만 변화를 알 수 있어,
반응이 늦어지고 중요한 사건을 지나친다.

## 목표

- [ ] 사용자가 자신과 관련된 주요 사건을 앱 안 한곳에서 모아 확인할 수 있다.
- [ ] 읽지 않은 알림이 있음을 사용자가 즉시 인지한다.

## 비목표

- 이메일·모바일 푸시 등 **앱 외부 채널로의 발송**은 이번 범위 밖이다.
- 사건 유형별 **구독 on/off 설정**은 이번 범위 밖이다(전체 사용자에게 동일 규칙).

## 요구사항

### 기능 요구사항

- [ ] 사용자와 관련된 지정 사건이 발생하면 그 사용자의 알림 목록에 새 항목이 생긴다.
- [ ] 각 알림은 **무슨 일이 · 언제 · 무엇에 대해** 일어났는지 사용자가 이해할 수 있는
      정보를 담는다.
- [ ] 사용자가 알림을 열면 읽음으로 표시되고 안읽음 개수에서 빠진다.
- [ ] 사용자가 알림에서 관련 대상으로 바로 이동할 수 있다.

### 비기능 요구사항

- [ ] 새 알림 발생 후 사용자가 인지하기까지의 지연이 **명시된 상한 이내**여야 하고 그 지연을
      관찰할 수 있다.
- [ ] 알림은 사용자별로 격리되어 **다른 사용자의 사건이 보이지 않는다**(권한).
- [ ] 알림이 많이 쌓여도 목록 조회가 **체감 지연 없이** 이루어진다.

## 수용 기준

- [ ] 지정 사건이 발생하면 대상 사용자의 목록에 항목이 나타나고, 대상이 아닌 사용자에게는
      나타나지 않는다.
- [ ] 안읽음이 하나라도 있으면 안읽음 표시가 보이고, 모두 읽으면 사라진다.
- [ ] 알림을 열면 읽음으로 바뀌고 안읽음 개수가 하나 줄어든다.
- [ ] 권한 밖 사용자의 사건은 목록에 **절대** 나타나지 않는다.

## 열린 질문

- 알림 보관 기간·최대 개수 정책은 무엇인가? (제품 결정 필요)
- 읽음 처리는 개별만인가, "모두 읽음"도 필요한가?

## ADR 필요 여부

- **필요:** 알림을 **어떻게 전달·저장하느냐**(사건 발생 시점에 미리 만들어 저장할지, 조회
  시점에 계산할지)가 되돌리기 어려운 구조 결정이고, "지연 상한·권한 격리·목록 성능"이라는
  요구를 만족시키는 방식에 대안이 여럿이다. → [`./notification-adr.md`](./notification-adr.md)에서
  선택지를 비교해 결정한다.

## 관련 문서

- ADR: [`./notification-adr.md`](./notification-adr.md)

## 고도 주해 (이 PRD가 왜 이 고도인가 — 학습용)

> 이 절은 예시 전용 설명이며, 실제 PRD에는 넣지 않는다. 각 요구가 왜 PRD 고도에 머물렀는지와,
> 흔히 새는 형태를 대조한다.

- **"각 알림은 무슨 일이·언제·무엇에 대해 일어났는지 담는다"** — 사용자가 이해해야 할 *정보의
  의도*만 규정한다. 여기에 `NotificationEntity.payload: jsonb`나 필드명을 적었다면 "고유명사
  소거 판정"에서 걸린다 — 그 이름을 지우면 요구가 무너지므로 구현 누수다. 컬럼·타입은 구현으로,
  저장 구조가 다투는 결정이면 ADR로 내린다.
- **"지연이 명시된 상한 이내여야 하고 관찰할 수 있다"** — *무엇이 만족돼야 하는가*를 관찰 가능한
  결과로 적었다. "15초마다 폴링한다"라고 적었다면 이는 전달 *방식*(구현/ADR)이지 요구가 아니다.
  구현이 폴링→푸시로 바뀌어도 이 요구 문장은 그대로 살아남는다("구현 결합 판정" 통과).
- **"알림이 많이 쌓여도 체감 지연 없이 조회된다"** — 성능을 *무엇*으로 적었다. "인덱스를 추가해
  100ms 내 응답" 같은 문장은 방법과 파라미터라 구현/ADR로 내린다.
- **비목표가 범위를 못 박는다** — 외부 채널·구독 설정을 명시적으로 제외해 scope creep을 막는다.
- **"ADR 필요 여부"가 결정을 미루지 않고 라우팅한다** — 전달·저장 구조는 대안이 여럿이고 되돌리기
  비싸므로 ADR 감이다. 이 판단만 PRD에 남기고, 실제 선택지 비교는 짝 ADR에서 한다.
- **수용 기준은 전부 통과/실패로 관찰 가능** — "좋은 알림 경험을 제공한다" 같은 검증 불가 문장이
  없다. 동시에 구현을 처방하지도 않아, 누수와 과소명세 양쪽을 피했다.
