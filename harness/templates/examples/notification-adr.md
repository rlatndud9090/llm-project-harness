# ADR: 알림 전달·저장 구조

## 컨텍스트

PRD([`./notification-prd.md`](./notification-prd.md))는 (1) 새 알림 인지까지의 지연 상한,
(2) 사용자별 권한 격리, (3) 알림이 쌓여도 저하되지 않는 목록 조회를 요구한다.

- **PRD 요구:** 위 세 요구를 만족시키는 알림 전달·저장 구조가 필요하다.
- **현재 제약:** 사건은 앱의 여러 지점에서 발생하고, 사용자별 목록 조회는 빈번하다.
- **되돌리기 비용:** 저장·전달 구조는 한번 자리 잡으면 데이터와 조회 경로가 얽혀 바꾸기 비싸다.

## 결정

**사건이 발생하는 시점에, 대상 사용자별 알림 레코드를 미리 만들어 저장한다**(사전 생성).
목록 조회는 저장된 레코드를 읽기만 한다.

## 선택지

### 선택지 A: 사건 시점 사전 생성

- 장점: 목록 조회가 단순·빠르다(요구 3 충족). 읽음/안읽음 상태를 레코드에 자연히 보관한다.
  권한 격리를 저장 시점에 확정한다(요구 2 충족).
- 단점: 저장 공간이 늘고, 사건 하나가 여러 사용자에게 퍼질 때 쓰기 비용이 커진다.

### 선택지 B: 조회 시점 계산

- 장점: 저장을 최소화한다.
- 단점: 매 조회마다 관련 사건을 모아 계산해야 해 목록 조회 성능이 요구 3과 충돌한다. 읽음
  상태를 별도로 보관해야 하고, 지연 상한(요구 1) 관리가 복잡해진다.

## 선택 근거

목록 조회가 빈번하고 지연 상한·조회 성능이 핵심 요구이므로, 조회를 단순·빠르게 만드는
**선택지 A**를 채택한다. 선택지 B는 저장을 아끼지만 조회 성능과 읽음 상태 관리에서 PRD 요구를
직접 위협하므로 기각한다.

## 결과

### 긍정적 영향

- 목록 조회가 단순해지고, 읽음/안읽음과 권한 격리가 저장 구조에서 자연히 보장된다.

### 부정적 영향 / 트레이드오프

- 저장 공간과 사건당 쓰기 비용이 늘어난다.
- 한 사건이 많은 사용자에게 퍼지는 경우(대량 팬아웃) 쓰기 부하가 집중된다.

## 후속 작업

- [ ] 보관 기간·최대 개수 정책 확정(PRD 열린 질문과 연결).
- [ ] 대량 팬아웃 사건의 쓰기 부하 완화 방안 검토(배치·비동기 등은 구현 단계 결정).

## 검증

- [ ] 지정 사건 발생 시 대상 사용자 목록에 항목이 나타나고, 대상이 아닌 사용자에겐 안 나타난다.
- [ ] 권한 밖 사용자의 사건이 목록에 나타나지 않는다.
- [ ] 알림이 대량으로 쌓인 사용자의 목록 조회가 지연 상한 내에 완료된다.

## 고도 주해 (이 ADR이 왜 이 고도인가 — 학습용)

> 이 절은 예시 전용 설명이며, 실제 ADR에는 넣지 않는다.

- **결정이 "사전 생성 vs 조회 시점 계산"이라는 대안의 선택**이다 — 되돌리기 비싸고 대안이 여럿인,
  전형적인 ADR 감이다. PRD는 이 결정을 "필요"로 라우팅만 했고, 실제 비교·선택은 여기서 한다.
- **테이블명·컬럼 타입·기술 스택·인덱스 전략은 여기에도 없다.** 그것들은 이 결정 아래의 *구현*이라
  코드·설계문서로 내려간다. ADR은 "어떤 구조를 왜 골랐나"까지만 남긴다("과대 ADR" 회피).
- **선택지 B를 형식적 들러리로 두지 않았다** — 실제 장단점과 기각 사유(조회 성능·읽음 상태 관리
  충돌)를 남겨, 다음 세션이 근거 없이 뒤집거나 맹종하지 않게 했다.
- **상태 필드가 불변성을 시연한다** — `status: accepted`, `supersedes` 자리. 이 결정이 나중에
  바뀌면 이 본문을 고치지 않고 새 ADR을 만들어 `supersedes`로 이 파일을 가리키고, 이 파일 상태를
  `superseded`로 바꾼다.
